# MISSION.md 格式

`MISSION.md` 位于工作区根目录,捕获用户学习本主题的**真实理由**。所有教学决策——下一步教什么、推送哪些资源、设计哪些练习——都应回溯到这份文档。

## 模板

```md
# 使命:{主题}

## 为什么
{1-3 句话。用户在现实世界中追求的具体目标。当他们掌握这项技能时,生活或工作会发生什么变化?避免"为了理解 X"这样的抽象表述——要追问底层的真实结果。}

## 成功是什么样
- {一项具体的、可观察到的能力}
- {另一项具体能力}
- {…}

## 约束条件
- {时间、预算、已有承诺、学习偏好,任何会限制方法的事实}

## 范围之外
- {用户明确表示暂时不追的相邻主题——保护最近发展区}
```

## 规则

- **一个工作区一个使命。** 如果用户想学两件不相关的事,那是两个工作区。
- **具体优于抽象。** "10 月前跑完半马"胜过"更健康";"给团队交付一个 Rust CLI"胜过"学 Rust"。
- **对含糊要追问。** 如果用户说不清为什么,先访谈再动笔。糟糕的使命比没有使命更糟。
- **现实变了就修订。** 使命会变。当用户目标变化时,更新本文件——别让过期的使命继续主导后续会话。
- **保持简短。** 如果 `MISSION.md` 超过一屏,它就不再是指南针,而变成了计划书。
