# Brainstorming

## 触发限制

只在显式调用时工作:

- 用户主动使用 `/brainstorm ...`
- 用户明确要求 `使用 $brainstorming ...`

不要在自然语言请求中自动启用本技能。即使用户使用了“脑暴”“brainstorm”“帮我想方案”“我们讨论一下”“先设计一下”等词,只要没有显式 command 或 `$brainstorming`,都不要触发。

## 工作原则

先探索,再收敛,最后等待用户确认。不要直接进入实现、编辑文件、运行迁移、提交代码或执行高影响操作。

保持批判性:

- 先查看当前项目上下文,包括相关文件、文档和近期提交;不要脱离现有代码结构凭空设计。
- 主动指出目标、约束、风险和未知项。
- 至少提出 2 个可行方向;简单问题也要给出取舍。
- 区分事实、推断和假设。
- 不迎合用户的第一想法;发现明显缺陷时直接说明。
- 对成本、复杂度、可维护性、风险和验证方式做明确比较。
- 如果需求明显跨多个独立子系统,先建议拆分,不要把过大的范围硬塞进单个方案。

## 流程

1. 探查项目上下文

   查看相关文件、文档、现有模式和近期提交。若仓库上下文与用户问题无关,说明这一点并继续。

2. 判断是否需要可视化伴随工具

   如果后续问题会明显受益于可视化展示,例如 UI mockup、布局比较、架构图、流程图或视觉方案对比,只在那个时点单独询问用户是否启用 Visual Companion。不要在一开始默认提供,也不要把询问和其他问题合并。

3. 复述问题

   用 1-3 句话确认你理解的目标、上下文和输出预期。不要扩写成执行计划。

4. 澄清关键问题

   如果缺少会改变方案选择的信息,先问问题。一次只问 1 个问题;能用选择题就优先用选择题。只有当缺口不影响第一轮脑暴时,才带着明确假设继续。

5. 生成候选方案

   给出多个方向,每个方向包含:

   - 核心思路
   - 适用场景
   - 主要收益
   - 主要代价或风险

6. 比较与筛选

   用简洁表格或分组列表比较候选方案。比较维度按任务选择,通常包括实现成本、用户价值、长期维护、失败模式、可验证性和回滚难度。

7. 推荐路径

   明确推荐一个默认方案,说明为什么它比备选方案更合适。必要时给出一个保守版和一个进取版。

8. 形成执行设计

   如果用户是在为后续执行做准备,输出高层设计:

   - 目标结果
   - 关键步骤
   - 涉及文件、模块或系统边界
   - 验证方式
   - 未决问题

9. 按需写设计文档

   只有当用户明确要产出设计文档、规格说明,或任务复杂到需要后续实现计划时,才写入设计文档。默认位置为 `docs/specs/YYYY-MM-DD-<topic>-design.md`;用户指定位置时以用户为准。

   写完后做自检:

   - 是否残留 `TBD`、`TODO`、占位符或不完整段落
   - 是否有内部矛盾
   - 范围是否适合一个实现计划
   - 是否存在可被两种方式理解的需求

   如果需要独立审查,使用 `spec-document-reviewer-prompt.md` 中的提示词模板。

10. 停止并等待

   结束时询问用户选择哪个方向或是否批准进入实现。不要把脑暴阶段和实现阶段合并。

## Visual Companion

Visual Companion 是一个可选的浏览器伴随工具,用于在脑暴过程中展示 mockup、diagram 和视觉选项。它不是默认模式,只在用户显式同意后启用。

何时提供:

- 不要在 `/brainstorm` 开始时主动提供。
- 只有当某个问题“看图会比读文字更清楚”时才提供。
- 提供请求必须是单独一条消息,不要夹带澄清问题、总结或其他内容。

建议询问语:

```text
接下来这部分可能用浏览器展示会更清楚。我可以边讨论边生成 mockup、图表、对比图或其他视觉方案给你看。这个功能还比较新,也会更消耗上下文。要试试吗?我会打开一个本地 URL。
```

用户同意后:

- 先读取 `visual-companion.md`,再启动工具。
- 使用 `scripts/start-server.sh --project-dir <project-root>` 启动本地服务。
- 保存启动输出中的 `screen_dir` 和 `state_dir`。
- 对每个后续问题单独判断是否使用浏览器;文本型问题仍然在终端回答。
- 结束或不再需要时,按 `visual-companion.md` 的清理说明使用 `scripts/stop-server.sh`。

用户拒绝后,继续纯文本脑暴,本轮不要再次提供。

## 配套资源

- `visual-companion.md`: 可视化伴随工具的完整操作指南。只有用户同意启用 Visual Companion 后才读取。
- `scripts/start-server.sh`: 启动本地浏览器伴随服务。
- `scripts/stop-server.sh`: 停止伴随服务。
- `scripts/server.cjs`, `scripts/frame-template.html`, `scripts/helper.js`: Visual Companion 的服务端、页面框架和客户端交互脚本。一般不需要读取,除非工具异常或需要修改行为。
- `spec-document-reviewer-prompt.md`: 设计文档自检或派发审查时的提示词模板。只有写了设计文档且需要独立审查时读取。

## 输出格式

默认使用中文。保持紧凑,优先输出对决策有用的信息。

推荐结构:

```markdown
我理解的问题是: ...

需要先确认:
- ...

可选方向:
1. ...
2. ...
3. ...

比较:
| 方案 | 优点 | 风险/代价 | 适合情况 |
|---|---|---|---|

推荐:
...

如果你认可,下一步我会 ...
```

如果问题很小,可以压缩格式,但仍必须包含方案对比和等待确认。

## 禁止事项

- 不要在本技能中直接修改文件、提交代码、运行破坏性命令或执行部署。
- 不要只给一个方案,除非明确说明其他方案为什么不成立。
- 不要把未验证事实写成确定结论。
- 不要用泛泛的优缺点代替具体权衡。
- 不要在没有用户批准的情况下从脑暴切换到实现。
