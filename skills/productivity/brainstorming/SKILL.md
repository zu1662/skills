# Brainstorming

## 触发限制

只在显式调用时工作:

- 用户主动使用 `/brainstorm ...`
- 用户明确要求 `使用 $brainstorming ...`

不要在自然语言请求中自动启用本技能。即使用户使用了"脑暴""brainstorm""帮我想方案""我们讨论一下""先设计一下"等词,只要没有显式 command 或 `$brainstorming`,都不要触发。

## 工作原则

先探索,再收敛,最后等待用户确认。不要直接进入实现、编辑文件、运行迁移、提交代码或执行高影响操作。

保持批判性:

- 先查看当前项目上下文,包括相关文件、文档和近期提交;不要脱离现有代码结构凭空设计。
- 主动指出目标、约束、风险和未知项。
- 至少提出 2 个可行方向;简单问题也要给出取舍。
- 区分事实、推断和假设。
- 不迎合用户的第一想法;发现明显缺陷时直接说明。
- 对成本、复杂度、可维护性、风险和验证方式做明确比较。
- 如果需求明显跨多个独立子系统,先建议拆分,不要把过大的范围硬塞进单个方案。

## 流程

1. 探查项目上下文

   查看相关文件、文档、现有模式和近期提交。若仓库上下文与用户问题无关,说明这一点并继续。

2. 判断是否需要可视化伴随工具

   如果后续问题会明显受益于可视化展示,例如 UI mockup、布局比较、架构图、流程图或视觉方案对比,只在那个时点单独询问用户是否启用 Visual Companion。不要在一开始默认提供,也不要把询问和其他问题合并。

3. 复述问题

   用 1-3 句话确认你理解的目标、上下文和输出预期。不要扩写成执行计划。

4. 澄清关键问题

   如果缺少会改变方案选择的信息,先问问题。一次只问 1 个问题;能用选择题就优先用选择题。只有当缺口不影响第一轮脑暴时,才带着明确假设继续。

5. 生成候选方案

   给出多个方向,每个方向包含:

   - 核心思路
   - 适用场景
   - 主要收益
   - 主要代价或风险

6. 比较与筛选

   用简洁表格或分组列表比较候选方案。比较维度按任务选择,通常包括实现成本、用户价值、长期维护、失败模式、可验证性和回滚难度。

7. 推荐路径

   明确推荐一个默认方案,说明为什么它比备选方案更合适。必要时给出一个保守版和一个进取版。

8. 形成执行设计

   如果用户是在为后续执行做准备,输出高层设计:

   - 目标结果
   - 关键步骤
   - 涉及文件、模块或系统边界
   - 验证方式
   - 未决问题

9. 用户确认整体设计

   结束脑暴阶段前,询问用户是否认可整体设计方向。不要在此步自动进入计划生成或实现。

   如果用户要求调整,回到步骤 5-8 迭代,直到用户明确认可。

10. 询问是否生成实现计划

    用户确认设计后,询问用户是否需要生成详细实现计划:

    ```text
    设计已确认。是否需要我生成详细实现计划?
    ```

    得到用户肯定答复后,进入"实现计划生成"流程(见下文)。如果用户拒绝,结束本次脑暴。

11. 询问执行方式并开始执行

    实现计划生成并通过自检后,询问用户选择哪种执行方式(见"执行交接")。得到用户选择后,开始执行。

## 按需写设计文档

只有当用户明确要产出设计文档、规格说明,或任务复杂到需要后续实现计划时,才写入设计文档。默认位置为 `docs/specs/YYYY-MM-DD-<topic>-design.md`;用户指定位置时以用户为准。

写完后做自检:

- 是否残留 `TBD`、`TODO`、占位符或不完整段落
- 是否有内部矛盾
- 范围是否适合一个实现计划
- 是否存在可被两种方式理解的需求

如果需要独立审查,使用 `spec-document-reviewer-prompt.md` 中的提示词模板。

## 实现计划生成

用户确认设计并同意生成计划后,使用以下方法生成详细实现计划。核心假设:执行者对当前代码库没有上下文,需要文档化所有必要信息——哪些文件要改、具体代码、测试方式、如何验证。给他们完整的、细粒度的任务。遵循 DRY、YAGNI、TDD、频繁提交原则。

**声明:** 开始生成时先告知"我正在生成详细实现计划。"

**保存位置:** `docs/plans/YYYY-MM-DD-<feature-name>.md`(用户指定位置时以用户为准)。

### 范围检查

如果规格说明覆盖多个独立子系统,建议拆分为多个独立的计划,每个计划应能独立产出可工作、可测试的软件。

### 文件结构

定义任务前,先映射哪些文件会被创建或修改,每个文件负责什么:

- 设计具有清晰边界和明确接口的单元,每个文件一个明确职责。
- 优先小而聚焦的文件,而非做太多事的大文件。
- 一起变更的文件应放在一起;按职责拆分,而非按技术层拆分。
- 在现有代码库中遵循既有模式。如果代码库使用大文件,不要单方面重构——但如果正在修改的文件已变得臃肿,在计划中包含拆分是合理的。

### 任务粒度控制

任务是最小单元,拥有独立的测试周期,值得一次独立的审查关卡。将初始化、配置、脚手架和文档步骤折叠到需要它们的任务中;只在审查者可能有意义地拒绝一个任务而批准相邻任务时才拆分。每个任务以一个可独立测试的交付物结束。

### 细粒度任务步骤

每一步是一个动作(2-5 分钟):

- "写失败测试" - 一步
- "运行确认它失败" - 一步
- "实现最小代码使测试通过" - 一步
- "运行测试确认通过" - 一步
- "提交" - 一步

### 计划文档头部

每个计划必须以以下头部开头:

```markdown
# [功能名称] 实现计划

**目标:** [一句话描述构建什么]

**架构:** [2-3 句关于方法的描述]

**技术栈:** [关键技术/库]

## 全局约束

[规格说明中项目级要求——版本下限、依赖限制、命名和复制规则、平台要求——每条一行,精确值从规格说明中逐字复制。每个任务的需求隐式包含此节。]

---
```

### 任务结构

每个任务遵循以下结构:

````markdown
### 任务 N: [组件名称]

**文件:**
- 创建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py:123-145`
- 测试: `tests/exact/path/to/test.py`

**接口:**
- 依赖: [此任务使用的前序任务的产物——精确签名]
- 产出: [后续任务依赖的产物——精确函数名、参数和返回类型。任务的执行者只看到自己的任务;这个块是他们了解相邻任务使用的名称和类型的方式。]

- [ ] **步骤 1: 写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2: 运行测试确认失败**

运行: `pytest tests/path/test.py::test_name -v`
预期: 失败,提示 "function not defined"

- [ ] **步骤 3: 写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4: 运行测试确认通过**

运行: `pytest tests/path/test.py::test_name -v`
预期: 通过

- [ ] **步骤 5: 提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

### 禁止占位符

每一步必须包含执行者需要的实际内容。以下都是**计划失败**——绝不写:

- "TBD"、"TODO"、"以后实现"、"补充细节"
- "添加适当的错误处理" / "添加验证" / "处理边缘情况"
- "为以上写测试"(没有实际测试代码)
- "类似任务 N"(重复代码——执行者可能乱序阅读任务)
- 描述做什么但不展示怎么做的步骤(代码步骤必须有代码块)
- 引用任何任务中都未定义的类型、函数或方法

### 自检

写完完整计划后,用全新的眼光对照规格说明检查计划。这是你自己运行的检查清单,不是派发子代理。

**1. 规格覆盖:** 浏览规格说明中每个章节/需求。能否指出实现它的任务?列出任何遗漏。

**2. 占位符扫描:** 搜索计划中的危险信号——上述"禁止占位符"中的任何模式。修复它们。

**3. 类型一致性:** 后续任务中使用的类型、方法签名和属性名是否与早期任务中定义的匹配?任务 3 中叫 `clearLayers()` 但任务 7 中叫 `clearFullLayers()` 就是 bug。

发现问题就原地修复。不需要复审——修了继续。如果发现规格需求没有对应任务,补上任务。

## 执行交接

计划生成并通过自检后,提供执行选择:

```text
计划已完成并保存到 docs/plans/<filename>.md。两种执行方式:

1. 逐任务执行(推荐) - 每个任务独立执行,任务间审查,快速迭代
2. 批量执行 - 在当前会话中批量执行任务,带检查点审查

选择哪种方式?
```

用户选择后开始执行:

- **逐任务执行:** 每个任务执行前重读该任务内容,执行完毕后运行测试验证,再进入下一任务。任务间向用户报告进度。
- **批量执行:** 按检查点批量执行任务,每个检查点暂停向用户报告进度。

执行原则:

- 严格按计划中的步骤顺序执行,不跳步。
- 每步完成后验证(运行测试、检查预期输出)。
- 遇到计划中未预见的问题时暂停,向用户报告,不擅自偏离计划。
- 频繁提交,每个任务结束时提交。

## Visual Companion

Visual Companion 是一个可选的浏览器伴随工具,用于在脑暴过程中展示 mockup、图表和视觉选项。它不是默认模式,只在用户显式同意后启用。

何时提供:

- 不要在 `/brainstorm` 开始时主动提供。
- 只有当某个问题"看图会比读文字更清楚"时才提供。
- 提供请求必须是单独一条消息,不要夹带澄清问题、总结或其他内容。

建议询问语:

```text
接下来这部分可能用浏览器展示会更清楚。我可以边讨论边生成 mockup、图表、对比图或其他视觉方案给你看。这个功能还比较新,也会更消耗上下文。要试试吗?我会打开一个本地 URL。
```

用户同意后:

- 先读取 `visual-companion.md`,再启动工具。
- 使用 `scripts/start-server.sh --project-dir <project-root>` 启动本地服务。
- 保存启动输出中的 `screen_dir` 和 `state_dir`。
- 对每个后续问题单独判断是否使用浏览器;文本型问题仍然在终端回答。
- 结束或不再需要时,按 `visual-companion.md` 的清理说明使用 `scripts/stop-server.sh`。

用户拒绝后,继续纯文本脑暴,本轮不要再次提供。

## 配套资源

- `visual-companion.md`: 可视化伴随工具的完整操作指南。只有用户同意启用 Visual Companion 后才读取。
- `scripts/start-server.sh`: 启动本地浏览器伴随服务。
- `scripts/stop-server.sh`: 停止伴随服务。
- `scripts/server.cjs`, `scripts/frame-template.html`, `scripts/helper.js`: Visual Companion 的服务端、页面框架和客户端交互脚本。一般不需要读取,除非工具异常或需要修改行为。
- `spec-document-reviewer-prompt.md`: 设计文档自检或派发审查时的提示词模板。只有写了设计文档且需要独立审查时读取。

## 输出格式

默认使用中文。保持紧凑,优先输出对决策有用的信息。

推荐结构:

```markdown
我理解的问题是: ...

需要先确认:
- ...

可选方向:
1. ...
2. ...
3. ...

比较:
| 方案 | 优点 | 风险/代价 | 适合情况 |
|---|---|---|---|

推荐:
...

如果你认可,下一步我会 ...
```

如果问题很小,可以压缩格式,但仍必须包含方案对比和等待确认。

## 禁止事项

- 不要在未经用户逐步确认的情况下进入计划生成或执行阶段。必须依次获得:设计确认 → 计划生成确认 → 执行方式选择。
- 不要在脑暴阶段直接修改文件、提交代码、运行破坏性命令或执行部署。
- 不要只给一个方案,除非明确说明其他方案为什么不成立。
- 不要把未验证事实写成确定结论。
- 不要用泛泛的优缺点代替具体权衡。
- 不要在计划中使用占位符或描述性步骤,每步必须有可执行的实际内容。
