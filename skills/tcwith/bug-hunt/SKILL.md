# Bug Hunt Swarm

## 核心原则

以“诊断优先、只读调查”为目标排查问题。先收集最小有效证据包，再从多个角度并行验证假设，最后由主代理汇总、排序根因并给出最快证明路径。

本技能默认不修改代码、不加临时埋点、不提交修复。只有用户在后续明确要求“修复”时，才退出本技能的只读边界并进入实现流程。

## 1. 构建 Bug 信息包

先收集最小可用信息，不因信息不全而停滞。优先使用用户已经提供的材料，其次读取仓库中最接近问题的证据。

信息包应包含：

1. 现象：哪里坏了，用户看到什么。
2. 预期：本来应该发生什么。
3. 实际：现在发生什么。
4. 复现：步骤、频率、环境、输入数据；未知时明确标注未知。
5. 影响范围：页面、接口、模块、平台、用户群、版本或分支。
6. 证据：日志、堆栈、失败用例、截图、监控告警、最近 diff、配置、运行环境。

证据来源优先级：

1. 用户描述、报错、截图、日志、测试输出。
2. 用户点名的文件、commit、分支、PR、模块。
3. 当前工作区变更、最近提交历史、相关测试。
4. 与失败路径最近的代码、配置、路由、状态管理、接口契约或构建脚本。

如果报告过于模糊，先推断一个“最小问题陈述”，并列出仍未知的信息。只有缺少关键入口导致无法开始时，才向用户追问。

## 2. 设定调查边界

在深入前写出简短调查摘要，作为所有调查员的共享背景：

1. 目前看起来坏掉的行为。
2. 已知证据和证据位置。
3. 尚未证明的假设。
4. 最可能相关的系统区域。
5. 什么证据可以确认或排除根因。

优先读取项目说明与局部约束：

- `AGENTS.md`、`README.md`、开发/测试说明。
- 受影响模块附近的架构、状态、路由、schema、接口或运行时文档。
- 与现象相关的最近 diff、commit、配置变更。

允许的只读动作：

- 使用 `rg`、`git diff`、`git log`、`git show`、`git blame`、`git status` 查证。
- 阅读日志、堆栈、配置、代码、测试和文档。
- 运行最小安全复现命令或已有测试命令，前提是不修改源码和外部系统。

禁止动作：

- 不编辑文件。
- 不写入临时诊断代码或新埋点。
- 不修复、不重构、不格式化无关文件。
- 不对生产系统执行有副作用的命令。

## 3. 启动四路只读调查

当问题足够复杂、跨模块、证据冲突或存在多种可能根因时，启动四个只读调查员并行工作。若当前工具或用户授权不允许创建子代理，则在主线程中按同样四个视角依次调查，并明确说明使用了降级流程。

四个调查视角：

1. 代码路径调查员：沿着失败入口追踪调用链、状态流、数据流、异常处理和边界条件。
2. 回归调查员：检查最近 diff、commit、配置、依赖、feature flag、环境差异，寻找“从何时开始坏”的证据。
3. 测试与复现调查员：分析失败测试、复现步骤、fixture、mock、异步时序、缓存和 flaky 条件。
4. 契约与运行时调查员：核对 API/schema、配置、构建产物、平台差异、权限、网络、时间、并发和外部依赖。

分派给每个调查员的任务必须包含：

- 同一份调查摘要。
- 该调查员的专属问题。
- 只读边界。
- 需要返回的证据格式。

调查员返回格式：

```markdown
## 结论
- 最可能根因：
- 置信度：高 / 中 / 低

## 关键证据
- `文件或命令输出位置`：证据说明

## 排除项
- 已检查但不像根因的方向：

## 最快验证路径
- 用什么最小步骤确认或证伪：

## 修复方向
- 不改代码，仅描述可能的修复位置和思路：
```

## 4. 交叉验证与排序

主代理汇总四路调查结果，不要简单拼接。必须检查证据之间是否互相支持、冲突或只是相关但不足以证明。

根因排序规则：

1. 直接解释用户现象和失败证据。
2. 与时间线、最近变更或环境差异吻合。
3. 有代码路径、日志、测试或配置证据支撑。
4. 能被一个小步骤快速确认或证伪。
5. 不依赖过多未验证假设。

置信度标准：

- 高：有直接证据，能解释现象，且可快速复现或已有失败输出支撑。
- 中：证据链合理，但缺少一个关键确认步骤。
- 低：只是可疑方向，证据不足或有替代解释。

如果多个方向都可能成立，保留前 2-3 个候选，不要强行给唯一结论。

## 5. 输出根因报告

最终答复使用中文，面向需要快速决策的开发者。先给最可能结论，再给证据和下一步。

推荐结构：

```markdown
**最可能根因**
一句话说明根因、影响路径和置信度。

**证据**
- `文件:行号` 或命令输出：说明为什么支持该结论

**候选根因排序**
1. 根因 A：置信度，支持证据，缺口
2. 根因 B：置信度，支持证据，缺口

**最快验证路径**
1. 运行/检查/复现某个最小步骤
2. 预期看到什么可以确认，看到什么可以排除

**建议修复方向**
- 描述应该改哪里、为什么，不直接修改代码

**仍未知**
- 需要用户补充或后续验证的信息
```

报告要求：

- 使用具体文件、函数、配置、commit 或命令输出作为证据。
- 明确区分“已证明”“合理推断”“待验证”。
- 不把没有证据的猜测包装成结论。
- 给出最快确认路径，而不是泛泛建议“多加日志”。
- 若未能定位根因，说明已排除什么、下一步最值得查什么。

## 6. 何时降级为单线程调查

遇到以下情况时，不强行启动并行调查：

- 用户没有明确授权使用子代理或并行代理。
- 当前环境没有可用的子代理工具。
- Bug 很小，单文件或单条堆栈即可定位。
- 信息不足到无法给不同调查员分配独立问题。

降级时仍保留四个视角，但压缩执行：

1. 先查代码路径。
2. 再查最近变更。
3. 再查测试/复现。
4. 最后查契约/运行时。

输出时说明“未启动并行子代理，已按四个调查视角完成只读排查”。

## 完成检查

交付前确认：

- 已构建 Bug 信息包或说明缺失项。
- 已遵守只读边界。
- 已检查项目局部说明和相关代码路径。
- 已至少从两个角度验证最可能根因，或说明为什么做不到。
- 已给出根因排序、证据、置信度和最快验证路径。
- 未在用户未要求时修改文件或实现修复。

## 示例提示词

- “用 bug-hunt 帮我排查这个线上报错的根因，先不要改代码。”
- “这个测试最近变 flaky 了，帮我并行分析可能原因并给最快验证路径。”
- “分析这个回归是哪个 commit 引起的，输出证据和候选根因排序。”
- “为什么这个页面打开白屏？请只读调查并给诊断报告。”
