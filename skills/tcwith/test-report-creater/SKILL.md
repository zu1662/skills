# 提测报告生成器

## 技能描述

基于 Git 分支和 commit key 生成面向测试人员的代码变更提测报告。该技能自动分析代码改动，提取关键信息，并生成结构化的测试报告，帮助测试人员快速了解改动范围和测试重点。

## 使用场景

当用户需要生成提测报告时，会提供以下信息：

- Git 分支名称
- Commit key 或 commit hash

如果没有提供分支名称，则以当前分支为准，比对分支为master（没有则使用release）分支。

## 工作流程

### 第一步：收集基础信息

1. **获取 Git 信息**
    - 使用 `git log` 获取 commit 信息（作者、时间、提交信息）
    - 使用 `git diff` 或 `git show` 获取代码变更详情
    - 识别变更的文件类型和数量

2. **提取关键信息**
    - 代码分支名称
    - 开发人员（commit 作者）
    - 提交时间
    - 文档生成时间（当前时间）
    - 文档生成人：提测助手

### 第二步：分析代码改动

1. **文件分类统计**
    - 统计各类型文件的变更数量（.js, .wxml, .wxss, .json 等）
    - 识别改动的模块或页面

2. **改动类型识别**
    - UI 改动：识别 .wxml, .wxss 文件的变更
    - 逻辑改动：识别 .js 文件中的函数、流程变更
    - 配置改动：识别 .json 配置文件的变更
    - 工具/公共代码改动：识别公共模块的变更

3. **关键改动提取**
    - 提取 commit message 中的关键信息
    - 分析代码变更的影响范围
    - 识别需要特别关注的重要改动

### 第三步：生成报告内容

根据模板结构生成报告：

1. **基础信息**：填写分支、人员、时间等基本信息
2. **改动范围详解**：分类整理改动，每类用简洁的语言描述
3. **测试注意事项**：基于改动类型提供测试建议
4. **其他**：补充需要额外关注的信息

### 第四步：输出报告

生成符合模板格式的 Markdown 格式报告，并保存到指定位置。

## 生成规则

### 改动描述规范

1. **使用简洁清晰的语言**
    - 避免技术术语，使用业务语言
    - 示例：✅ "优化了订单提交流程" ❌ "重构了 submitOrder 方法"

2. **不包含具体代码**
    - 除非是需要特别关注的重要改动
    - 代码片段使用代码块格式标注

3. **改动类型分类原则**
    - UI 改动：页面布局、样式、交互元素的变更
    - 逻辑改动：业务流程、数据处理、状态管理的变更
    - 配置改动：配置文件、环境变量、参数调整
    - 公共代码改动：公共组件、工具函数、基础库的变更
    - 其他改动：不属于以上分类的变更

### 文件类型映射

| 文件类型 | 改动类型                |
| -------- | ----------------------- |
| .wxml    | UI 改动                 |
| .wxss    | UI 改动                 |
| .js      | 逻辑改动 / 公共代码改动 |
| .json    | 配置改动                |
| .wxs     | 逻辑改动                |

### 测试建议生成规则

1. **核心功能测试建议**
    - 基于改动类型推断测试重点
    - 结合 commit message 提取测试场景
    - 为每个测试点提供明确的测试场景和预期结果

2. **边界及回归测试建议**
    - 根据改动范围推荐回归模块
    - 基于改动类型提供边界测试场景
    - 识别可能影响的关联功能

## 执行指令

当用户提供分支和 commit 信息时：

1. 首先确认用户提供的分支名称和 commit key
2. 执行 Git 命令获取变更信息
3. 分析代码变更并分类
4. 按照模板生成报告
5. 询问用户是否需要保存报告到文件（默认保存路径：`.codebuddy/test-report-{timestamp}.md`）

## 示例对话

**用户**：帮我基于 `feature/login-optimization` 分支的最新 commit 生成提测报告

**助手**：

1. 执行 `git log feature/login-optimization -1` 获取最新 commit 信息
2. 执行 `git diff feature/login-optimization^..feature/login-optimization` 获取变更详情
3. 分析变更并生成报告
4. 输出完整的提测报告

## 注意事项

1. **确保 Git 仓库状态正常**：在生成报告前确保当前目录是 Git 仓库且分支信息正确
2. **时间格式统一**：使用 `YYYY-MM-DD HH:mm:ss` 格式
3. **改动分类不超过四类**：根据实际改动情况选择最相关的分类
4. **避免过度技术化**：报告面向测试人员，使用业务语言而非技术语言
5. **保持客观准确**：基于实际代码变更生成报告，不夸大或遗漏改动范围

## 文档生成

把报告内容写入到 `test-report-{YYYYMMDD}.md` 文件中
