# Wiki Fetcher

## 作用

这个 skill 用于从企业内部 Wiki 系统中提取已登录可见的文档内容，并将其整理为结构清晰、可直接复用的 Markdown 输出。

它适合以下场景：

- 用户提供 Wiki 链接，希望自动登录后提取正文
- 页面需要先完成 OAuth 认证，认证完成后才能读取内容
- 页面正文位于 `id="tiptap"` 的编辑器容器中
- 输出需要保留标题层级、列表、段落与图片
- 输出末尾需要补充当前文档的核心要点总结

默认目标站点为 `https://wiki.17u.cn/`，认证完成的最低判定条件是浏览器当前地址已经稳定回到该域名下的 Wiki 页面。

## 何时使用

在用户出现以下意图时使用：

- “帮我抓取这个 wiki 页面内容”
- “把内部 wiki 转成 Markdown”
- “登录公司 wiki 后提取正文”
- “读取 tiptap 编辑器里的内容并整理输出”
- “总结这篇 wiki 文档”

## 推荐输入

优先接受以下输入：

- Wiki 文档链接，例如 `https://wiki.17u.cn/wiki?fid=8daecbf42d304ebab7314a468eaa3b41`
- 图片处理偏好：保留原图链接，或下载后再引用
- 是否只要正文，还是需要附加“核心要点总结”

如果用户只给出模糊描述，没有提供链接，先要求补充链接。

## 输入要求

用户应尽量提供：

- 可访问的 Wiki 链接，例如 `https://wiki.17u.cn/wiki?fid=...`
- 是否允许打开浏览器完成 OAuth 认证（默认允许）
- 是否需要保留原图链接，或将图片下载后再引用（默认保留原图链接）

如果用户没有提供链接，先要求补充链接，再继续。

## 工作流程

### 1. 校验链接与抓取目标

1. 检查用户提供的链接是否是目标 Wiki 域名。
2. 如果不是企业 Wiki 链接，明确说明当前 skill 只适用于目标 Wiki 系统，并请求正确链接。
3. 记录目标页面地址，后续用于判断 OAuth 是否已完成。
4. 默认提取目标是该链接所对应页面中的正文区域，而不是附件页、评论区或接口返回。

完成标准：

- 已拿到合法的 Wiki 页面链接
- 已确认抓取目标是页面正文而不是附件、跳转页或接口地址

### 2. 触发 OAuth 认证

1. 优先使用系统浏览器访问用户提供的 Wiki 链接；如果当前运行环境只能使用内置浏览器，先明确告知限制，再请求用户确认是否继续。
2. 等待用户完成 OAuth 登录流程。
3. 在等待阶段，只观察页面跳转、当前地址和可见状态，不要立即读取正文节点或查询页面数据。
4. 反复检查当前页面地址是否已经稳定回到目标 Wiki 域名。
5. 只有当页面地址稳定停留在目标 Wiki 域名，且不再继续跳转时，才视为认证成功。
6. 认证成功后，再开始后续正文定位与内容提取。

决策分支：

- 如果浏览器工具只能打开内置页面而不是系统浏览器，先说明限制，再请求用户确认是否继续使用当前浏览器能力。
- 如果认证后页面仍停留在统一登录页、错误页或空白页，不继续提取，先报告当前状态。
- 如果页面发生多次跳转，以最终稳定停留在 Wiki 域名页面为准。
- 如果地址短暂回到 Wiki 域名后又继续跳走，仍视为认证未完成，继续等待。
- 如果等待超时仍未回到目标 Wiki 域名，停止抓取并要求用户检查账号权限、网络或重新登录。

等待规则：

- 在 OAuth 完成前，不允许查找 `#tiptap`
- 在 OAuth 完成前，不允许提取任何正文文本、图片或页面数据
- 只有“地址稳定回到 Wiki 域名”这个条件成立后，才能进入下一步

完成标准：

- 已确认 OAuth 成功
- 页面可见且不是重定向中间页
- 页面地址已稳定，不再继续跳转

### 3. 定位正文容器

1. 仅在 OAuth 完成后，在最终页面中查找 `#tiptap`。
2. 如果页面存在 iframe，先定位包含正文的 iframe，再查找 `#tiptap`。
3. 如果未找到 `#tiptap`，检查是否存在延迟渲染，再进行一次等待与重试。
4. 只从 `#tiptap` 内部提取正文，忽略站点导航、顶部工具栏、评论区、悬浮目录和操作按钮。

决策分支：

- 如果重试后仍未找到 `#tiptap`，停止并向用户报告页面结构与失败原因。
- 如果页面有多个可疑正文节点，优先选择 `#tiptap`，不要猜测其他节点替代。

完成标准：

- 已定位到唯一的正文容器
- 确认容器中存在可读取的文本或图片内容

### 4. 提取正文与图片

1. 按页面真实顺序提取正文内容。
2. 保留以下结构信息：
   - 标题层级
   - 普通段落
   - 有序和无序列表
   - 引用块
   - 代码块
   - 表格（如页面结构中可识别）
   - 图片
3. 对图片优先提取可直接访问的 `src` 或最终渲染地址。
4. 如果图片链接是相对路径，将其解析为绝对地址。
5. 根据用户偏好决定是保留原图链接，还是先下载图片再在 Markdown 中引用本地资源。
6. 不要丢失正文中的图片位置顺序。

质量要求：

- 不要擅自改写原文意思
- 不要把导航栏、评论区、工具栏、目录浮层混入正文
- 不要丢失图片与相邻说明文字的对应关系
- 不要把图片 alt 文本、说明文字和正文段落重新排序

### 5. 生成 Markdown

1. 将提取结果转换为 Markdown。
2. 保持内容顺序与结构尽量贴近原文。
3. 输出时保证可读性，避免纯粹逐节点堆砌。
4. 图片使用标准 Markdown 图片语法。
5. 如果某些富文本结构无法等价表达，优先选择最接近语义的 Markdown 表达方式。
6. 如果页面标题可可靠识别，将其作为 Markdown 一级标题；否则不要臆造标题。

格式要求：

- 标题使用标准 Markdown 标题语法
- 段落之间保留合理空行
- 列表缩进清晰
- 代码块使用 fenced code block
- 图片单独成行，必要时保留原始说明文字

### 6. 总结核心要点

1. 在 Markdown 正文之后增加“核心要点总结”章节。
2. 用精炼条目总结文档主旨、关键步骤、重要约束和风险点。
3. 总结应基于已提取正文，不得编造未出现的信息。
4. 如果文档主要是流程说明，优先总结前置条件、操作步骤、异常分支和产出结果。

总结要求：

- 优先提炼结论、流程、限制条件
- 如果原文信息不完整，要明确标注“原文未说明”
- 不要输出空泛总结

## 输出格式

最终输出应包含两个部分：

1. 整理后的 Markdown 正文
2. “核心要点总结”章节

推荐输出骨架：

```markdown
# 文档标题

正文内容...

![图片说明](https://...)

## 核心要点总结

- 要点 1
- 要点 2
- 要点 3
```

如果抓取失败，输出应包含：

- 失败发生在哪一步
- 当前页面状态或关键错误信息
- 建议用户如何补充信息或重试

## 完成检查

在交付前逐项确认：

- 已使用用户提供的 Wiki 链接
- 已确认 OAuth 完成
- 已从 `#tiptap` 中提取正文
- 已按 Markdown 输出文本与图片
- 已保留原文主要结构与顺序
- 已附加核心要点总结
- 未混入无关页面元素

## 护栏

- 不要在未完成认证时声称可以访问正文
- 不要在 OAuth 等待阶段提前读取 DOM 正文或执行内容提取
- 不要在未找到 `#tiptap` 时凭空生成正文内容
- 不要把页面 chrome、菜单、评论区、按钮文案当作正文
- 不要编造图片链接、缺失段落或摘要结论
- 当工具能力无法满足“系统浏览器”要求时，必须先告知限制
